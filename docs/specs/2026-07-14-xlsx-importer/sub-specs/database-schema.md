# Database Schema

This is the database schema implementation for the spec detailed in @docs/specs/2026-07-14-xlsx-importer/spec.md

The importer writes to tables defined in `@docs/specs/2026-07-14-core-data-model/sub-specs/database-schema.md`.
It adds two of its own. These are the only schema changes this spec introduces.

## import_runs

```sql
CREATE TABLE import_runs (
  id            integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  file_name     varchar(260) NOT NULL,
  file_sha256   char(64)     NOT NULL,
  started_at    timestamptz  NOT NULL,
  finished_at   timestamptz  NULL,
  outcome       varchar(10)  NOT NULL CHECK (outcome IN ('Running', 'Succeeded', 'Failed', 'Aborted')),
  rows_read     integer      NOT NULL DEFAULT 0,
  rows_written  integer      NOT NULL DEFAULT 0,
  rows_skipped  integer      NOT NULL DEFAULT 0,
  anomaly_count integer      NOT NULL DEFAULT 0,
  sheet_summary jsonb        NULL,
  error_message text         NULL,

  CONSTRAINT ck_import_runs_finished_iff_terminal
    CHECK ((outcome = 'Running') = (finished_at IS NULL))
);

CREATE INDEX ix_import_runs_started ON import_runs (started_at DESC);
```

`file_sha256` answers "which version of the workbook produced this data?" — a question that becomes urgent the
moment the 83,000 mi row is corrected in Excel and the import re-run. Without the hash, two runs are
indistinguishable.

`sheet_summary` is `jsonb` holding per-sheet counts (`{"Fuel Log": {"read": 13, "written": 13, "skipped": 0}}`).
A child table would be more normal, but this is a write-once audit blob that is never queried relationally, and
`jsonb` keeps the run atomic.

`rows_read`/`rows_written`/`rows_skipped` are counters on an append-only audit row, not derived values — they
describe an event that has finished, and no query recomputes them. This is not a DEC-002 violation.

The `Running` outcome exists so a crashed import is distinguishable from one that never started.

## data_anomalies

```sql
CREATE TABLE data_anomalies (
  id            integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  import_run_id integer      NULL REFERENCES import_runs(id) ON DELETE SET NULL,
  vehicle_id    integer      NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,

  kind          varchar(24)  NOT NULL CHECK (kind IN (
                  'MileageNonMonotonic',
                  'FuelCostDiscrepancy',
                  'SupersededByMirror',
                  'ImplausibleMpg',
                  'UnparseableValue',
                  'MissingReference')),
  severity      varchar(8)   NOT NULL CHECK (severity IN ('Error', 'Warning', 'Info')),

  entity_type   varchar(40)  NOT NULL,
  entity_id     integer      NULL,
  source_sheet  varchar(40)  NULL,
  source_row    integer      NULL,

  message       text         NOT NULL,
  detail        jsonb        NULL,

  status        varchar(10)  NOT NULL CHECK (status IN ('Open', 'Accepted', 'Corrected', 'Dismissed')),
  resolved_at   timestamptz  NULL,
  resolution_note text       NULL,

  created_at    timestamptz  NOT NULL,
  updated_at    timestamptz  NOT NULL,
  source        varchar(8)   NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed')),

  CONSTRAINT ck_anomalies_resolved_iff_terminal
    CHECK ((status IN ('Open')) = (resolved_at IS NULL))
);

CREATE INDEX ix_anomalies_vehicle_status ON data_anomalies (vehicle_id, status, severity);
CREATE INDEX ix_anomalies_entity ON data_anomalies (entity_type, entity_id);
```

### Why `(entity_type, entity_id)` here and not on `documents`

The core data model spec rejects polymorphic links for `documents` in favour of three real FKs, because README
§3.9 names exactly three link targets. The reasoning inverts here: an anomaly can attach to *any* entity, and
some attach to a row that was deliberately **not** imported — the lumped fuel expense has no `entity_id` at all,
only a sheet and row number. A real FK cannot point at a row that does not exist. `entity_id` is therefore
nullable, and `source_sheet`/`source_row` carry the provenance instead.

Consistency with `documents` would cost more than it buys. The difference is real, not accidental.

### Why anomalies have a lifecycle

`status` is not decoration. The 83,000 mi row will sit `Open` until the owner decides — and the decision has
three legitimate outcomes:

- `Corrected` — it was 80,300; the reading is fixed and the anomaly records what changed.
- `Accepted` — the odometer really did read 83,000 (a swapped cluster, say); the data stands and the flag stops
  nagging.
- `Dismissed` — not worth pursuing.

Without a lifecycle, the anomaly report is either permanently noisy or gets cleared by deletion, which destroys
the record that the data was ever questioned. `resolution_note` is where the reasoning lives.

`severity` separates "this is wrong" (`Error` — non-monotonic mileage) from "this is expected but you should
know" (`Info` — the superseded lumped row).

### Anomaly rows this import must produce

Against the real workbook at reference date 2026-07-14:

| Kind | Severity | Expected |
|---|---|---|
| `MileageNonMonotonic` | Error | Exactly 1 — Service History, 27 Jun 2026, 83,000 mi against a current 80,712 |
| `SupersededByMirror` | Info | Exactly 1 — Expenses Log lumped fuel row, £725.70 |

Both counts are asserted by tests. `FuelCostDiscrepancy` and `ImplausibleMpg` counts are unknown until the
importer runs against the real file; whatever they are becomes the recorded baseline, and a later change to that
count is a regression.

## Migration

These two tables ship as a second migration (`AddImportTracking`) rather than being folded into
`InitialSchema`. The core schema is what the application needs; import tracking is what a one-off migration
needs. Keeping them separate means the core migration can be reviewed against README §2 without import
machinery mixed in.
