# Database Schema

This is the database schema implementation for the spec detailed in @docs/specs/2026-07-14-core-data-model/spec.md

Target: PostgreSQL 17. All tables created by a single initial migration. SQL below is the intended shape —
EF Core generates it from the configurations; it is written out here so the shape can be reviewed before any
code exists.

## Conventions applied to every table

- `id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- Vehicle-scoped tables carry `vehicle_id integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE`
- Mutable tables carry the audit block:

```sql
created_at timestamptz NOT NULL,
updated_at timestamptz NOT NULL,
source     varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
```

- `notes text NULL CHECK (notes <> '')` wherever notes appear — null means absent, empty string is never valid
- Money is `numeric(10,2)`, mileage is `integer`, no floats anywhere

## Reference tables

Global, not vehicle-scoped. Seeded closed; editable in settings per README §2.

```sql
CREATE TABLE expense_categories (
  name         varchar(24) PRIMARY KEY,
  display_order integer    NOT NULL,
  is_system    boolean     NOT NULL DEFAULT false
);

CREATE TABLE garages (
  name     varchar(80) PRIMARY KEY,
  contact  varchar(120) NULL,
  address  text         NULL,
  notes    text         NULL CHECK (notes <> '')
);

CREATE TABLE wash_locations (
  name  varchar(80) PRIMARY KEY,
  notes text        NULL CHECK (notes <> '')
);
```

`is_system` marks the 13 categories from README §2 that the domain reasons about by name (notably `Fuel`, which
the auto-mirroring in §3.2 depends on). System categories may be renamed for display but not deleted — deleting
`Fuel` would silently break fuel-to-expense mirroring.

## vehicles

Static reference per README §2. The insurance, breakdown, fluid, and tyre blocks are EF owned types mapped to
prefixed columns on this table rather than separate tables — they are 1:1 with the vehicle and have no
independent lifetime.

```sql
CREATE TABLE vehicles (
  id                       integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,

  -- identity
  registration             varchar(16)  NOT NULL,
  make                     varchar(40)  NOT NULL,
  model                    varchar(60)  NOT NULL,
  variant                  varchar(40)  NULL,
  year                     integer      NOT NULL,
  colour                   varchar(30)  NULL,
  body_style               varchar(30)  NULL,
  vin                      varchar(17)  NULL,

  -- purchase
  purchase_date            date         NOT NULL,
  seller                   varchar(120) NULL,
  purchase_price           numeric(10,2) NULL,
  purchase_mileage         integer      NOT NULL,

  -- engine / drivetrain
  engine_code              varchar(30)  NULL,
  engine_size_cc           integer      NULL,
  fuel_type                varchar(12)  NOT NULL CHECK (fuel_type IN ('Petrol', 'Diesel', 'Hybrid', 'Electric', 'LPG')),
  transmission             varchar(30)  NULL,
  drivetrain               varchar(30)  NULL,

  -- fluids and specs
  oil_spec                 varchar(60)  NULL,
  oil_capacity_litres      numeric(4,2) NULL,
  coolant_spec             varchar(60)  NULL,
  coolant_capacity_litres  numeric(4,2) NULL,
  brake_fluid_spec         varchar(40)  NULL,
  transmission_oil_spec    varchar(60)  NULL,
  spark_plug_part          varchar(40)  NULL,
  oil_filter_part          varchar(40)  NULL,
  air_filter_part          varchar(40)  NULL,
  fuel_filter_part         varchar(40)  NULL,
  cabin_filter_part        varchar(40)  NULL,

  -- tyres
  tyre_size                varchar(24)  NULL,
  pressure_front_psi       numeric(4,1) NULL,
  pressure_rear_psi        numeric(4,1) NULL,
  pressure_front_laden_psi numeric(4,1) NULL,
  pressure_rear_laden_psi  numeric(4,1) NULL,
  min_tread_mm             numeric(3,1) NULL,

  -- lifecycle (DEC-007)
  status                   varchar(6)   NOT NULL DEFAULT 'Active' CHECK (status IN ('Active', 'Sold', 'SORN')),
  is_default               boolean      NOT NULL DEFAULT false,

  -- statutory
  mot_expiry_seed          date         NULL,
  ved_annual_cost          numeric(10,2) NULL,
  ved_expiry               date         NULL,
  ulez_compliant           boolean      NULL,

  -- insurance (owned block)
  insurance_insurer        varchar(80)  NULL,
  insurance_policy_number  varchar(60)  NULL,
  insurance_period_start   date         NULL,
  insurance_period_end     date         NULL,
  insurance_cover_type     varchar(40)  NULL,
  insurance_premium        numeric(10,2) NULL,
  insurance_excess_compulsory numeric(10,2) NULL,
  insurance_excess_voluntary  numeric(10,2) NULL,
  insurance_ncb_years      integer      NULL,

  -- breakdown cover (owned block)
  breakdown_provider       varchar(80)  NULL,
  breakdown_policy_number  varchar(60)  NULL,
  breakdown_expiry         date         NULL,

  -- default garage
  default_garage           varchar(80)  NULL REFERENCES garages(name) ON DELETE SET NULL,

  notes                    text         NULL CHECK (notes <> ''),
  created_at               timestamptz  NOT NULL,
  updated_at               timestamptz  NOT NULL,
  source                   varchar(8)   NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

-- registration_normalized varchar(16) GENERATED ALWAYS AS (upper(replace(registration, ' ', ''))) STORED
CREATE UNIQUE INDEX ix_vehicles_registration ON vehicles (registration_normalized);
CREATE UNIQUE INDEX ix_vehicles_default ON vehicles (is_default) WHERE is_default;
```

*(Revised during implementation: a stored generated column with a unique index, rather than an expression
index — EF Core cannot model expression indexes, and the generated column is equivalent.)*

The partial unique index allows at most one `is_default = true` row — the MCP fallback target when a tool
call names no vehicle (DEC-007). Zero defaults is legal (an empty garage); two is not.

`status` gates presentation, not computation: a Sold vehicle keeps its history and its metrics still compute —
the garage and attention surfaces simply stop featuring it.

**`mot_expiry_seed` is not the MOT expiry.** Derived MOT expiry is `MAX(next_due_date)` over `service_records`
where `type = 'MOT'`; this column is read only when no such record exists. Naming it `mot_expiry` would invite
exactly the defect being designed out — the sheet's stale 6 Aug 2026 against a real 8 Jul 2027.

The unique index normalises case and spacing so `BT53AKJ` and `BT53 AKJ` cannot both exist.

## mileage_readings

The decoupling entity per README §2. Current mileage derives from here, never from a manual field.

```sql
CREATE TABLE mileage_readings (
  id         integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  reading_date date  NOT NULL,
  mileage    integer NOT NULL CHECK (mileage >= 0),
  origin     varchar(10) NOT NULL CHECK (origin IN ('manual', 'fuel', 'tyre', 'wash', 'service')),
  notes      text    NULL CHECK (notes <> ''),
  created_at timestamptz NOT NULL,
  updated_at timestamptz NOT NULL,
  source     varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_mileage_readings_vehicle_date ON mileage_readings (vehicle_id, reading_date DESC);
CREATE INDEX ix_mileage_readings_vehicle_mileage ON mileage_readings (vehicle_id, mileage DESC);
```

`origin` (which log produced the reading) is distinct from `source` (which surface wrote it). A fill-up logged
via MCP is `origin = 'fuel'`, `source = 'mcp'`. Both matter and neither substitutes for the other.

Two indexes because "latest reading" is ambiguous in this data: README §2 says current mileage is the
"max/most-recent" reading, and the workbook proves those can disagree — the 27 Jun 2026 service row logs
83,000 mi against a current 80,712. Ordering by date and ordering by mileage return different rows. The
derived-metrics spec resolves which wins; both access paths are indexed so that decision stays cheap.

**No monotonicity `CHECK` constraint.** A constraint would reject the import outright. README §5.3 requires
flagging anomalies, not refusing them.

## expense_entries

```sql
CREATE TABLE expense_entries (
  id           integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id   integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  entry_date   date    NOT NULL,
  category     varchar(24) NOT NULL REFERENCES expense_categories(name) ON DELETE RESTRICT,
  sub_category varchar(60) NULL,
  vendor       varchar(120) NULL,
  amount       numeric(10,2) NOT NULL,
  mileage      integer NULL CHECK (mileage >= 0),
  payment_method varchar(30) NULL,
  fuel_entry_id  integer NULL UNIQUE REFERENCES fuel_entries(id) ON DELETE CASCADE,
  notes        text    NULL CHECK (notes <> ''),
  created_at   timestamptz NOT NULL,
  updated_at   timestamptz NOT NULL,
  source       varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_expense_entries_vehicle_date ON expense_entries (vehicle_id, entry_date DESC);
CREATE INDEX ix_expense_entries_vehicle_category ON expense_entries (vehicle_id, category);
```

`fuel_entry_id` is the mirroring link required by README §3.2. It is `UNIQUE` so a fill can mirror to at most
one expense, and `ON DELETE CASCADE` so deleting a fill removes its mirror rather than orphaning it. This link
is what closes the £163.16 gap: the workbook carries one lumped "fuel to date" expense row instead of per-fill
entries, so Dashboard fuel YTD (£725.70) and the Fuel Log (£888.86) disagree. With mirroring, they are the
same rows.

No running-total column. The workbook's is a formula over ~30 trailing blank rows; the replacement is `SUM()`.

`ON DELETE RESTRICT` on category prevents deleting a category still in use.

## fuel_entries

```sql
CREATE TABLE fuel_entries (
  id             integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id     integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  entry_date     date    NOT NULL,
  mileage        integer NOT NULL CHECK (mileage >= 0),
  litres         numeric(6,3) NOT NULL CHECK (litres > 0),
  price_per_litre numeric(6,3) NOT NULL CHECK (price_per_litre > 0),
  total_cost     numeric(10,2) NOT NULL CHECK (total_cost > 0),
  station        varchar(80) NULL,
  fill_level     varchar(8) NOT NULL CHECK (fill_level IN ('Full', 'Half', 'Quarter')),
  notes          text    NULL CHECK (notes <> ''),
  created_at     timestamptz NOT NULL,
  updated_at     timestamptz NOT NULL,
  source         varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_fuel_entries_vehicle_date ON fuel_entries (vehicle_id, entry_date DESC);
CREATE INDEX ix_fuel_entries_vehicle_mileage ON fuel_entries (vehicle_id, mileage DESC);
```

**No `mpg`, `l_per_100km`, or `miles_since_last` columns.** All three derive from this row and its predecessor.
Storing them is what produced the doubled litres total (1,112.94 against a real 556.47 — exactly 2.0000×,
because the summary counted all 13 fills twice).

`total_cost` alongside `litres × price_per_litre` is deliberate and is not a stored derived value — it is
transcribed from the receipt, which is authoritative. Forecourt rounding routinely makes the product differ by
a penny. No `CHECK` enforces agreement; the write path flags a discrepancy over 2p as an anomaly.

`fill_level` matters because MPG is only meaningful between two full-to-full fills. A partial fill makes the
computed figure for that interval unreliable, and the derived-metrics service must know to say so.

## service_records

```sql
CREATE TABLE service_records (
  id             integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id     integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  service_date   date    NOT NULL,
  mileage        integer NOT NULL CHECK (mileage >= 0),
  type           varchar(40) NOT NULL,
  garage         varchar(80) NULL REFERENCES garages(name) ON DELETE SET NULL,
  work_done      text    NULL,
  parts_replaced text    NULL,
  cost           numeric(10,2) NULL,
  next_due_date  date    NULL,
  next_due_mileage integer NULL CHECK (next_due_mileage >= 0),
  notes          text    NULL CHECK (notes <> ''),
  created_at     timestamptz NOT NULL,
  updated_at     timestamptz NOT NULL,
  source         varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_service_records_vehicle_date ON service_records (vehicle_id, service_date DESC);
CREATE INDEX ix_service_records_type_next_due ON service_records (vehicle_id, type, next_due_date DESC);
```

`type` is free text rather than an enum — the workbook's Service History uses varied descriptions and forcing
them into a closed list would lose information on import. But `type = 'MOT'` is load-bearing: it is how derived
MOT expiry is found. The importer normalises MOT rows to exactly that literal, and the third index serves that
lookup.

This table holds the 83,000 mi row dated 27 Jun 2026 — above the current 80,712, and likely 80,300 mistyped. It
imports as-is, with an anomaly flag raised against it. Silently correcting it would destroy the evidence; a
`CHECK` would reject the import.

## check_definitions and check_logs

```sql
CREATE TABLE check_definitions (
  id             integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id     integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  name           varchar(80) NOT NULL,
  cadence_label  varchar(40) NOT NULL,
  interval_days  integer NOT NULL CHECK (interval_days > 0),
  guidance       text    NULL,
  display_order  integer NOT NULL,
  is_active      boolean NOT NULL DEFAULT true,
  created_at     timestamptz NOT NULL,
  updated_at     timestamptz NOT NULL,
  source         varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE UNIQUE INDEX ix_check_definitions_vehicle_name ON check_definitions (vehicle_id, name);

CREATE TABLE check_logs (
  id                  integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  check_definition_id integer NOT NULL REFERENCES check_definitions(id) ON DELETE CASCADE,
  performed_on        date    NOT NULL,
  result              varchar(12) NULL CHECK (result IN ('OK', 'Attention', 'Failed')),
  notes               text    NULL CHECK (notes <> ''),
  created_at          timestamptz NOT NULL,
  updated_at          timestamptz NOT NULL,
  source              varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_check_logs_definition_date ON check_logs (check_definition_id, performed_on DESC);
```

`check_logs` is scoped through its definition rather than carrying its own `vehicle_id` — the definition is
already vehicle-scoped and a second path would allow the two to disagree.

**No `next_due` or `status` column.** Both derive from the latest log plus `interval_days`. A check with zero
logs is not "overdue" and not "OK" — it is *never logged*, a real fourth state. The workbook has 18 check rows
but its Dashboard counts 17, because "Spare tyre pressure" has never been logged and silently falls out of the
buckets. The absence of a status column is what forces the derived-metrics service to handle the empty case
explicitly instead of defaulting it.

`is_active` allows retiring a check without deleting its history.

## maintenance_tasks

```sql
CREATE TABLE maintenance_tasks (
  id             integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id     integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  kind           varchar(8) NOT NULL CHECK (kind IN ('DIY', 'Workshop')),
  priority       varchar(6) NOT NULL CHECK (priority IN ('High', 'Medium', 'Low')),
  title          varchar(160) NOT NULL,
  description    text    NULL,
  estimated_cost numeric(10,2) NULL,
  status         varchar(12) NOT NULL CHECK (status IN ('Open', 'InProgress', 'Scheduled', 'Done')),
  target_date    date    NULL,
  target_service varchar(80) NULL,
  completed_date date    NULL,
  assigned_garage varchar(80) NULL REFERENCES garages(name) ON DELETE SET NULL,
  service_record_id integer NULL REFERENCES service_records(id) ON DELETE SET NULL,
  notes          text    NULL CHECK (notes <> ''),
  created_at     timestamptz NOT NULL,
  updated_at     timestamptz NOT NULL,
  source         varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed')),

  CONSTRAINT ck_tasks_garage_workshop_only
    CHECK (assigned_garage IS NULL OR kind = 'Workshop'),
  CONSTRAINT ck_tasks_completed_date_iff_done
    CHECK ((status = 'Done') = (completed_date IS NOT NULL))
);

CREATE INDEX ix_tasks_vehicle_status ON maintenance_tasks (vehicle_id, status, priority);
```

Named `maintenance_tasks` / `MaintenanceTask` to avoid colliding with `System.Threading.Tasks.Task` in every
async method that touches it.

Single table with a `kind` discriminator per README §2 — not EF TPH inheritance. DIY and Workshop share every
field except `assigned_garage`, enforced nullable-and-Workshop-only by constraint.

`service_record_id` supports README §3.3's one-click promotion of a completed workshop task into a service
record, and preserves the link afterwards.

`ck_tasks_completed_date_iff_done` makes the two directions agree — a Done task without a date, or a completed
date on an Open task, are both incoherent.

Five PSI columns and four tread columns per README §2 — spare has a pressure but no tracked tread. All nullable:

## documents

```sql
CREATE TABLE documents (
  id             integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vehicle_id     integer NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  type           varchar(20) NOT NULL CHECK (type IN ('V5C', 'Insurance', 'MOT', 'Receipt', 'Photo', 'Manual', 'Other')),
  title          varchar(160) NOT NULL,
  document_date  date    NULL,
  file_path      varchar(400) NOT NULL,
  content_type   varchar(100) NOT NULL,
  size_bytes     bigint  NOT NULL CHECK (size_bytes > 0),
  sha256         char(64) NULL,

  service_record_id integer NULL REFERENCES service_records(id) ON DELETE SET NULL,
  expense_entry_id  integer NULL REFERENCES expense_entries(id) ON DELETE SET NULL,
  issue_id          integer NULL REFERENCES issues(id) ON DELETE SET NULL,

  notes          text    NULL CHECK (notes <> ''),
  created_at     timestamptz NOT NULL,
  updated_at     timestamptz NOT NULL,
  source         varchar(8)  NOT NULL CHECK (source IN ('web', 'mcp', 'import', 'seed'))
);

CREATE INDEX ix_documents_vehicle_type ON documents (vehicle_id, type);
```

`file_path` per DEC-005 — files on a mounted volume, path in the DB. Stored relative to a configured root so the
mount point can move without a data migration.

Three nullable FKs rather than a polymorphic `(entity_type, entity_id)` pair, because README §3.9 names exactly
three link targets. Real FKs get real referential integrity; a polymorphic pair gets none. `ON DELETE SET NULL`
keeps the document when its linked record goes — the MOT certificate outlives the service row.

`sha256` supports duplicate detection and backup verification. Nullable because it is not yet computed on
import.

## Seed data

Inserted by the initial migration with `source = 'seed'`. **Global reference data only** (revised 2026-07-14,
DEC-007):

1. **Expense categories** — the 13 from README §2: Fuel, Service, Repair, Parts, Insurance, Tax, MOT, Wash, Parking, Tools/Equipment, Breakdown, Purchase, Misc. All `is_system = true`.

That is the whole list. **Vehicles are never seeded, and neither is anything scoped to one.** The original
draft also seeded the BT53 AKJ vehicle, its 18 check definitions, garages, and wash locations — which collided
with the importer, which creates the vehicle from the Vehicle Info sheet, the checks from the Regular Checks
sheet, and upserts garages and wash locations from side columns. With the unique registration index, importing
into a seeded database would fail; without it, everything would double. One creator per row:

- **Vehicles** — created by the importer or the add-car flow, never by a migration.
- **Check definitions** — imported with their vehicle, or chosen at creation (empty / generic starter set /
  copy from an existing vehicle). The generic starter set is a code constant in `CarTracker.Domain` — template
  input to the add-car flow, not data with a lifecycle, so it gets no table.
- **Garages / wash locations** — upserted by the importer from side columns, editable in settings.

The BT53 AKJ facts (2003 Freelander 1, 1.8 SE K-series, purchased 14 Mar 2026 at 76,632 mi, coolant **OAT
red/pink — never IAT**) therefore live in the importer's expectations and the add-car flow's defaults, not in a
seed script.

## Rationale summary

| Choice | Reason |
|---|---|
| No derived columns anywhere | DEC-002; the four Dashboard defects are all stored-derived-value drift |
| Vehicles never seeded | DEC-007; the importer creates them, and a seed would collide with it on the registration index |
| `status` + partial-unique `is_default` | DEC-007; Sold/SORN keeps history out of the attention surfaces, and at most one vehicle answers unnamed MCP calls |
| `mot_expiry_seed`, not `mot_expiry` | Derived from the latest `type = 'MOT'` service record; the name prevents misuse |
